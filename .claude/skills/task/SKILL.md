# /task {номер} — Выполнение задачи из плана имплементации

Ты получаешь номер задачи как аргумент (например, `/task 2`). Выполни следующие шаги.

## 1. Прочитай план

Прочитай файл `docs/feature-branch/implementation-plan.md` с помощью Read.

## 2. Найди задачу

Найди секцию, соответствующую номеру задачи из аргумента. Секции имеют формат:

```
## N. Задача M: Название
```

где **M** — номер задачи из аргумента. Обрати внимание: порядковый номер секции (N) может отличаться от номера задачи (M).

Если задача с указанным номером не найдена — сообщи пользователю и останови выполнение.

## 3. Проверь зависимости

Посмотри граф зависимостей в начале плана. Если задача зависит от другой (например, задача 4 зависит от 3), проверь, что чекбоксы задачи-зависимости уже отмечены `[x]`. Если нет — предупреди пользователя и спроси, продолжать ли.

## 4. Войди в режим планирования

Вызови EnterPlanMode и спланируй выполнение задачи:

- Перечисли все файлы, которые нужно изменить (они указаны в секции «Файлы»)
- Прочитай каждый из этих файлов
- Составь пошаговый план изменений, основываясь на чекбоксах задачи
- Предложи план пользователю и дождись подтверждения

## 5. Выполни задачу

После подтверждения плана (пользователь подтвердит или вызовет ExitPlanMode):

- Выполни все пункты из чекбоксов задачи последовательно
- Для каждого изменения используй Edit/Write
- После завершения запусти проверки:
  ```bash
  npm run lint
  npm run format
  npm run build
  ```
- Если сборка или линтинг упали — исправь ошибки

## 6. Напиши отчёт

Выведи подробный отчёт о проделанной работе:

```
## Отчёт: Задача N — Название

### Что сделано
- Перечисли каждое изменение с указанием файла

### Изменённые файлы
- Список файлов с кратким описанием изменений

### Проверки
- npm run lint: ✅ / ❌
- npm run format: ✅ / ❌
- npm run build: ✅ / ❌
```

## 7. Обнови документацию

Сразу после отчёта обнови документацию:

1. **`docs/feature-branch/implementation-plan.md`** — отметь выполненные пункты: замени `- [ ]` на `- [x]` для каждого выполненного чекбокса задачи. Если изменения затронули чекбоксы других задач (например, часть работы задачи 4 выполнена в рамках задачи 3) — отметь и их.
2. **`docs/project-overview.md`** — если менялись типы в `domain.ts`, структура файлов или архитектурные решения, обнови соответствующие секции (доменная модель, структура проекта и т.д.).
3. **Другие `docs/*.md`** — обнови при необходимости (api-layer, ui-kit, coding-conventions и т.д.).

## 8. НЕ коммить

**НЕ выполняй** `git add`, `git commit`, `git push` или создание PR. Пользователь сделает это сам.
